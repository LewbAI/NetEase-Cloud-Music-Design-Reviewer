# 大厂AI 设计评审团 · 开源skill

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)
[![Next.js](https://img.shields.io/badge/Next.js-15-black)](https://nextjs.org)
[![TypeScript](https://img.shields.io/badge/TypeScript-5-blue)](https://www.typescriptlang.org)

上传一张或多张设计稿，5 位顶级大佬——"设计与商业大师"（乔纳森·伊夫、唐纳德·诺曼、张小龙、乔布斯、马斯克）当场评审、吵架、博弈，最后给你一个打了分的结论，并为你提供改进方向。

> 这套评判方法论（拉姆斯十大原则的实战应用 + "减法=自信"母法则）诞生于一线视觉审稿实践，这里开源出通用版本，供大家复用这套判断框架、替换成自己的评委阵容和标尺。

## 为什么做这个

设计团队把「审美」和「情感」当作核心竞争力，但真正判断设计好不好的能力，往往在少数人脑子里——不可复制、不可追溯、不可规模化。

AI 设计评审团做的事，就是把这套判断规则——拉姆斯十大原则、诺曼的认知模型——**显性化、代码化，让专业标准 7×24 在线**：即使没有专业设计师在场，也能拿到客观、可复现的评判；设计分歧也从"我觉得 vs 你觉得"，变成有依据、可追溯的理性博弈。

## 为什么开源

我们相信好设计值得被规模化，我们把这套判断框架通过开源交到更多人手里——你可以替换成自己的评委阵容、调整评分标尺、接入自己的设计原则，让它长成适合你团队的样子。

我们期待和大家一起共建：贡献新的评审视角、打磨提示词、补充设计理论、修 bug、提想法。欢迎 Star、Fork、提 Issue 和 PR——**一起把"好品味"变成人人可用的基础设施**。

## 大厂AI 设计评审团 · 全貌

想了解这套工具背后完整的评判逻辑（而不只是"怎么用"），看这一节。对应实现见 `lib/prompt.ts`。

### 一、架构设计（三层）

主席（AI）居中协调，底下三层叠加做判断：

- **第 1 层 · 隐形理论底座**（不展示）——内在思维框架，不直接出现在评委发言里。包含 Don Norman《设计心理学》、Nielsen 十大可用性启发、Shneiderman 八大黄金法则、Alan Cooper 的 Persona + Goal-Directed Design、John Maeda《简单法则》、Steve Krug 三铁律。**只用人话说，禁止在报告里引用"第几条原则"这类术语**。
- **第 2 层 · 团队设计法则**（客观对照标尺，可引用）——主标尺是 Dieter Rams 十大原则（权重 80–90%），背书参照 Apple HIG（10–20%）。违反时可以引用具体编号，如"违反 Rams 原则 10（尽可能少）"。
- **第 3 层 · 5 位专家评审团**（吵架可见）——每位评委用本人真实的口气、词汇、思维方式发声，互相博弈，不是套用抽象角色。

主席的角色：协调 5 位评委 + 内嵌的理论底座，最终给出一个最优解。报告顺序是**感性 → 减法 → 原则解释**，不是反过来当机器扫描。

### 二、5 位评委各看什么

| 评委 | 类别 | 关注点 | 口气 |
|---|---|---|---|
| Jony Ive | 设⁠计⁠专⁠家 | 圆角是否一致到 sub-pixel 级别、字体的 micro-typography、阴影/模糊/过渡的物理真实性、形体之间的呼吸节奏、颜色的"语义" | 温和但极度挑剔 |
| Don Norman | 设⁠计⁠专⁠家 | 可供性（Affordance）vs 意符（Signifier）、用户的概念模型是否与设计师一致、反馈延迟、状态可见性 | 教授型，爱举例对比 |
| 张小龙 | 业⁠务⁠专⁠家 | 这屏是否在打扰用户、哪些元素是"运营硬塞"的、存不存在"用户用完即走"的可能性、产品能否自己解释 | 冷静温和，直击本质 |
| Steve Jobs | 业⁠务⁠专⁠家 | 是不是放了太多东西、每个元素是否有理由存在、能不能砍掉 50% 内容后变得更好、有没有让用户"思考"的地方 | 粗暴直接，没有客气话 |
| Elon Musk | 业⁠务⁠专⁠家 | 这个元素删了会发生什么、这个流程是否必须存在、质疑"需求本身"是否合理、用户达成目标的速度 | 偏极端直接，爱用"删掉"、"第一性原理" |

### 三、评价维度（5 项叠加 = 100）

| 维度 | 满分 | 评估什么 |
|---|---|---|
| 减法·克制度 | 30 | 是否做到尽可能少，没有加法过度 |
| 调性·美观度 | 25 | 色彩、字体、形态语言整体调性 |
| 视觉层级 | 20 | 主次清晰、层级关系明确 |
| 完成度·细节 | 15 | 圆角、对齐、间距、像素级打磨 |
| 功能清晰度 | 10 | 用户能否看懂"该做什么" |

`score` 必须等于 5 个维度之和。核心母法则：**减法 = 自信，加法 = 不自信**——执行"心理删除测试"：对每个视觉元素问"删掉后这张图会变好、差不多、还是变差？"变好或差不多 → 该删；变差 → 元素挣到了位置，保留。

分段标签：

| 分数 | 标签 |
|---|---|
| 96–100 | 到位 |
| 90–95 | 基本到位（默认基准） |
| 80–89 | 可用 |
| 76–79 | 需要打磨 |
| 71–75 | 问题较多 |
| <70 | 严重不及格 |

### 四、评审流程

1. **基础扫描**（用第 1 + 2 层）算客观底分
2. **5 位评委各发一句话**（30–60 字，用本人口气）
3. **找冲突**：从 5 个发言里识别 1–2 处明显意见相反的点
4. **冲突展开**：让对立双方各反驳 1 次
5. **主席裁决**：综合所有视角，给出最优解

给得越全，评审越准——信息加分：PRD +3 / Persona +2 / 用户目标 +3 / 关心点 +2，全给齐封顶 +10。如果用户给了"关心点"，冲突展开里必须有一条围绕它。

三种模式：**单稿评审** / **双稿对比**（选出更好的一稿，不预设两稿的品牌归属）/ **新增模块评审**（双轨判断：新模块本身是否到位 + 与原页面是否兼容）。

语言硬性要求（中文模式下）：全程纯中文，即使是外国评委也说中文，专有名词要翻译成中文（如 Affordance → 可供性），只有人名和产品名可以保留原文。输出严格 JSON，不允许 markdown 围栏或额外文字。（右上角可切换英文模式，对应切换成全英文输出）

## 怎么用（最简单的方式）

**把整个文件夹拖给 Claude Code或者同类大模型，说一句"帮我启动"。**

1. 克隆 / 下载这个仓库
2. 把文件夹拖进 Claude Code或者同类大模型
3. 跟它说："**帮我启动**"
4. 跟着它的提示走（中间会要你给 Anthropic API Key或者同类大模型API key，没有的话它会教你 5 分钟拿一个）

完了。Claude Code 或者同类大模型会自动读 `CLAUDE.md`，一步步带你装好 + 启动。后续想用，再拖一次或双击 `start.command`。

### 手动启动

```bash
npm install
cp .env.local.example .env.local   # 把里面的占位符换成你自己的 Anthropic API Key
npm run dev
```

## 工作原理（大白话）

1. **你交稿**：上传设计稿，可选附上 PRD / 用户是谁 / 用户想干嘛 / 你最担心哪儿——给得越全评审越准
2. **系统先立标准**：默认采用迪特·拉姆斯十大原则（Dieter Rams' Ten Principles），搭上几位设计大师的经典理论，再用 Apple 人机界面指南补充——这套标尺可以替换成你自己团队的设计法则（见 `lib/prompt.ts`）
3. **5 位大师各自点评**：设计专家（乔纳森·伊夫、唐纳德·诺曼）+ 业务专家（张小龙、乔布斯、马斯克）各看一眼各说一句
4. **意见不合当场吵**：分歧点拎出来，对立双方各讲各的理，吵架过程看得到
5. **主席拍板**：揉成一个最终结论 + 分数 + 怎么改

## 三种模式

| Tab | 输入 | 输出 |
|---|---|---|
| 单稿评审 | 1 张设计稿 | 评分 + 总结 + 好在哪 + 待提升 + 维度表 + 评审席 + 吵架 |
| 双稿对比 | 2 张候选稿 | 胜出方 + 逐维度对比 + 评审席 + 吵架 |
| 新增模块评审 | 原页面 + 1~2 个新增方案 | 新模块评价 + 兼容性 + 评审席 + 吵架 |

报告右上角可「保存为图片」（自带被审设计稿）或「复制 Markdown」导出。

## 自定义

- **换评委阵容 / 评判标准**：改 `lib/prompt.ts`，里面的 5 位评委人设、Rams 十大原则权重、分数体系都是明文 prompt，随便改
- **换设计法则**：把 Rams 十大原则换成你团队自己的设计规范即可，架构不用动

## 注意

- 中国大陆需开代理（ClashX/Surge 等），否则 Anthropic API 会 403
- `.env.local` 里有你的 API Key，**不要提交到 git / 不要分享给任何人**（已加入 `.gitignore`）
- 本项目仅供个人 / 内部使用参考，上传的设计稿图片默认存在本地 `data/` 和 `public/uploads/`，不会上传到任何第三方（除了发去 Anthropic API 做评审）
